메모리 최적화에도 불구하고, 로컬 캐시는 애플리케이션 서버에 부담이 매우 크며, 배포마다 캐시가 초기화되며 추가적인 비용이 드는 캐싱 웜업 문제가 있었습니다. 또한 서버가 확장되는 경우에 일관되지 않는 캐시 문제도 예상되었습니다.

💡 의사결정 과정

장기적인 관점에서 확장성과 데이터 일관성을 확보하기 위해 Redis 기반의 캐시 아키텍처로 전환을 결정했습니다. 또한 캐시 쓰기 서버를 애플리케이션 서버와 분리하여 애플리케이션의 메모리 부담을 줄였습니다.

✅ 결과

애플리케이션 서버의 메모리 부담을 50% 이상 절감하여 EC2 인스턴스 비용 효율을 높였고, 모든 서버가 항상 일관된 캐시 데이터를 사용하게 되어 데이터 정합성 문제를 원천적으로 해결했습니다.

이 경험을 통해 기술 도입의 트렌드를 맹목적으로 따르기보다, 우리 서비스가 직면한 문제의 본질과 비즈니스 우선순위를 먼저 파악하는 것이 중요함을 배웠습니다. 긴급도, 자원, 확장성 등 주어진 조건 속에서 최적의 해결책을 찾아나가는 문제 해결 능력을 기를 수 있었습니다.

📊 문제 상황 분석

외부 API(OpenAI, Claude)의 갑작스러운 장애로 인해 TIL 태그 및 레퍼런스 데이터가 유실되는 치명적인 문제가 발생했습니다. 단일 외부 시스템에 의존하는 구조는 99.9% 가용성 요구사항을 충족할 수 없었습니다.

💡 의사결정 과정

외부 시스템 장애 시에도 서비스 연속성을 보장하기 위해 Chain of Responsibility 패턴을 적용했습니다. 실패 시 다음 외부 시스템을 순차적으로 호출하는 구조로 설계하여 단일 실패 지점을 제거했습니다.

public String callAIWithSimpleFallback(
    List<Map<String, Object>> messages,
    Map<String, Object> functionDefinition
) {
    for (AIClient client : aiClients) {
        try {
            log.info("Attempting to call {} API", client.getClientName());
            String result = client.callAI(messages, functionDefinition);
            log.info("Successfully called {} API", client.getClientName());
            return result;

        } catch (Exception e) {
            log.warn("Failed to call {} API: {}", client.getClientName(), e.getMessage());
            if (isLastClient(client)) {
                throw new RuntimeException("All AI services failed", e);
            }
            log.info("Trying next AI service");
        }
    }

    throw new RuntimeException("No available AI services");
}

⚡ 4-2: 재시도 큐를 통한 최종 일관성 보장
📊 새로운 문제 발견

Failover 구조로도 모든 외부 시스템이 동시에 장애가 발생할 수 있었습니다. 이 경우 사용자 데이터가 영구적으로 유실될 위험이 있었고, 일시적 장애 후 복구 시에도 데이터 정합성을 보장할 수 없었습니다.

💡 의사결정 과정

모든 외부 시스템이 실패할 경우를 대비하여 스케줄링 기반 재시도 큐 재처리 아키텍처를 설계했습니다. 실패한 이벤트를 큐에 저장하고 @Scheduled 기반의 재시도 로직으로 최종적 일관성(Eventual Consistency)을 보장했습니다.

✅ 결과

외부 시스템의 일시적 장애에도 데이터 유실을 완전히 방지하고, 시스템 복구 후 자동으로 데이터 정합성을 복원할 수 있게 되었습니다.
