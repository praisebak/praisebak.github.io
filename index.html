프로젝트 상세 소개

우아한테크코스 7기에서 진행 중인 프로젝트로, 조직단위로 열리는 이벤트를 놓치지 않게 해주는 알람 리마인더 플랫폼입니다. 대규모 트래픽 대응과 시스템 안정성 확보를 위한 기술적 도전과제를 해결하며, 우테코 내 크루,코치를 대상으로 서비스하고 있습니다.

개발 인원

백엔드 4명, 프론트엔드 3명

기술 스택

Java, Spring boot, JPA, MySQL, Flyway, Docker, Github Actions, Nginx, AWS, CloudWatch, WAF

  • 대규모 트래픽 대응을 위한 처리율 제한기(Rate Limiter) 구현 및 성능 최적화
  • Scale-Out 아키텍처 적용으로 RTS 600 달성

문제 해결

대규모 트래픽 대응 처리율 제한기 및 Scale-out 아키텍쳐 도입

문제 상황: 11만 건의 요청으로 인한 서비스 지연

아맞다 프로젝트 데모 데이에서 메인 화면 API가 11만 건의 요청을 받으며, 전체 페이지 로딩 속도가 1~2초까지 느려지는 현상이 발생했습니다. Grafana 대시보드 분석 결과, DB나 CPU 부하는 낮았지만 스레드 풀이 과도하게 할당되는 것을 확인했습니다. 이는 리소스 병목보다는, 순간적으로 몰린 대량의 요청으로 인한 애플리케이션의 동시 처리 능력 한계가 원인이었습니다.

🚀 1단계: 서비스 보호를 위한 처리율 제한기(Rate Limiter) 도입

💡 문제 정의 및 의사결정

데모 데이에서 발생한 11만 건의 대규모 요청으로 API 응답 지연이 확인되었습니다. Grafana 분석 결과, DB나 CPU 부하보다는 과도한 스레드 할당이 원인으로, 이는 악의적인 트래픽으로부터 서비스 보호가 필요함을 시사했습니다. 여러 대안 검토 후, JWT로 사용자를 식별할 수 있는 서비스 특성을 활용해 사용자 기반의 처리율 제한기를 도입하는 것이 가장 효과적이라고 판단했습니다.

💡 구현: In-Memory 슬라이딩 윈도우

추가 인프라 비용 없이 빠른 구현을 위해, Spring Filter를 이용한 In-Memory 방식의 슬라이딩 윈도우 알고리즘을 직접 구현했습니다. '최근 1분간 100회 초과 요청'을 비정상 트래픽으로 간주하고, HTTP 429 상태 코드와 함께 요청을 차단하도록 설계했습니다.

처리율 제한기 코드

✅ 검증: DoS 공격 상황 시뮬레이션

JMeter를 사용하여 단일 사용자가 대량의 요청을 보내는 부하 테스트를 진행했습니다. 처리율 제한기는 설정된 임계치를 초과하는 요청의 99.9%를 성공적으로 차단했으며, 서버 부하가 감소하여 유효 요청의 평균 응답 속도는 오히려 50% 개선되는 효과를 확인했습니다.

⚡ 2단계: 정상 트래픽 처리를 위한 Scale-Out 아키텍처 적용

📊 새로운 과제 발견

처리율 제한기는 비정상 트래픽 방어에는 효과적이었으나, 다수의 정상 사용자가 동시에 접속하는 상황을 가정한 부하 테스트에서는 평균 응답 속도가 5초 이상으로 급증하며 처리 용량의 한계를 보였습니다. 이에 애플리케이션 최적화를 넘어 인프라 확장으로 해결 방향을 전환했습니다.

💡 아키텍처 변경: Nginx 로드 밸런서 도입 및 인스턴스 확장

기존 Nginx에 로드 밸런싱 역할을 추가하고, 동일한 애플리케이션 인스턴스를 총 3대로 증설하여 수평 확장(Scale-Out)을 구현했습니다. Nginx의 upstream 모듈을 사용하여 들어오는 요청을 여러 인스턴스로 분산시켜 전체 처리량을 증대시켰습니다.

실제 멤버 300명 설정, 부하 테스트 결과

✅ 최종 결과 및 학습

인스턴스 확장 후, 300명의 동시 사용자를 가정한 부하 테스트에서 평균 응답 속도 0.3초를 달성했으며, 실제 런칭 데이의 대규모 트래픽을 안정적으로 처리했습니다. 이 경험을 통해 트래픽 규모에 맞는 인프라 확장 전략의 중요성을 체감했습니다.

다만 현재의 In-Memory 처리율 제한기는 인스턴스별로 개별 동작하는 한계가 있음을 인지하고 있으며, 다음 단계로 Redis 기반의 분산 처리율 제한기로 전환하여 아키텍처를 더욱 발전시킬 계획입니다.

힙 덤프 분석을 통한 대규모 데이터 처리 메모리 최적화

최적화 작성글

문제 상황: OutOfMemory(OOM) 발생 위기

태그 연관관계를 분석하여 캐시에 저장하는 과정에서, 메모리 사용량이 200MB에서 700MB까지 급증하는 문제가 발생했습니다. 이는 EC2 프리티어 인스턴스 가용 메모리의 약 80%를 점유하는 심각한 수준으로, 서비스의 잠재적인 OOM 장애 위험을 안고 있었습니다.

🚀 1단계: 대용량 데이터 처리 과정의 메모리 누수 해결

💡 원인 분석: Hibernate 1차 캐시

VisualVM을 이용한 힙 덤프(Heap Dump) 분석 결과, Hibernate 관련 객체들이 메모리의 대부분을 차지하고 있음을 확인했습니다. 이는 단일 트랜잭션 내에서 수만 건의 데이터를 처리하는 동안, Hibernate의 1차 캐시(영속성 컨텍스트)가 모든 엔티티를 계속 유지하며 메모리 누수와 유사한 현상을 일으키는 것이 핵심 원인이었습니다.

💡 해결 과정: 배치 처리와 영속성 컨텍스트 관리

이를 해결하기 위해, 전체 작업을 작은 단위로 나누어 처리하는 배치(Batch) 방식을 도입했습니다. 각 배치 작업이 끝날 때마다 entityManager.flush()와 entityManager.clear()를 명시적으로 호출하여, 1차 캐시를 비워 메모리를 즉시 해제하도록 구현했습니다. 이 과정에서 발생한 LazyInitializationException 문제는, 초기 데이터 조회 시 FETCH JOIN을 사용하여 연관관계를 미리 로딩하는 방식으로 해결했습니다.

✅ 1차 개선 결과

배치 처리 도입 후, 메모리 사용량이 급증했다가 급격히 해제되는 안정적인 톱니파(Sawtooth) 형태의 그래프를 확인할 수 있었습니다. 이를 통해 캐시 생성 과정에서의 OOM 발생 위험을 원천적으로 제거했습니다.

⚡ 2단계: 캐시 데이터 구조 자체의 메모리 최적화

💡 추가 분석 및 해결 과정

1단계 해결 후에도 최종적으로 생성된 캐시 객체가 약 250MB의 메모리를 차지하는 것을 확인했습니다. 원인은 도메인 특성을 고려하지 않은 Java Collection의 기본 초기 용량(Initial Capacity) 설정 때문이었습니다. '하나의 게시글은 최대 5개의 태그를 가진다'는 도메인 정책에 근거하여, 캐시 데이터 구조 내의 ArrayList와 HashMap의 초기 용량을 최적값으로 설정하여 불필요한 메모리 예약을 최소화했습니다.

✅ 최종 결과

데이터 구조 최적화를 통해 캐시 객체의 메모리 사용량을 250MB에서 200MB로 약 50MB(20%) 추가 절감했습니다. 이 경험을 통해 대용량 데이터를 다룰 때 Hibernate 영속성 컨텍스트에 대한 깊은 이해와 도메인 지식을 기반으로 한 자료구조의 최적화가 얼마나 중요한지 배울 수 있었습니다.

캐싱 웜업 문제 해결을 위한 분산 캐시(Redis) 마이그레이션


상황 & 문제: 로컬 캐시의 운영 한계 봉착

이전 단계에서 도입한 로컬 캐시는 응답 속도를 개선했지만, 운영 환경에서 다음과 같은 명확한 한계를 드러냈습니다.

  • 배포 시 데이터 유실: 애플리케이션 서버와 캐시의 생명주기가 동일하여, 배포할 때마다 모든 캐시 데이터가 초기화되는 문제가 발생했습니다.
  • 메모리 부담 및 장애 위험: 캐시를 생성(Warm-up)하는 과정에서 애플리케이션 메모리를 추가로 300MB나 사용했습니다.(초기 애플리케이션 메모리 사용량 100MB) 이는 저사양(EC2 t2.micro) 서버 가용 메모리의 상당 부분을 차지하여, OOM 장애의 직접적인 원인이 될 수 있었습니다.

🚀 해결 과정: 기술 스택 선정 및 아키텍처 분리

💡 의사결정: 분산 캐시 도입 결정

애플리케이션 자체의 최적화만으로는 한계가 명확했습니다. 근본적인 문제인 '캐시 쓰기' 작업의 부하를 해결하기 위해, 캐시의 생명주기를 애플리케이션과 분리하고 메모리 부담을 이전할 수 있는 분산 캐시 도입을 결정했습니다.

💡 기술 비교 및 최종 선택: Redis

미래 확장성과 개발 편의성을 고려하여 여러 기술을 비교 검토했습니다.

  • Memcached: Spring 진영의 공식 지원 부재로 인한 수동 설정의 번거로움, 복제(Replication) 기능 미지원으로 인한 가용성 문제 때문에 제외했습니다.
  • Caffeine: 고성능 로컬 캐시지만, '배포 시 데이터 유실'이라는 핵심 문제를 해결할 수 없어 제외했습니다.
  • Redis: Spring Cache 추상화를 완벽하게 지원하여 코드가 깔끔하고, 분산 캐시로서의 모든 요구사항을 만족하며, 분산 락 등 추후 확장성까지 뛰어나 최종적으로 채택했습니다.
💡 아키텍처 설계

시스템의 역할을 명확히 분리하여 안정성을 높이는 아키텍처를 설계했습니다.

  • 메인 애플리케이션 서버: 사용자의 요청 처리 및 캐시 조회(Read)만 담당.
  • 캐시 쓰기 서버: 주기적으로 데이터를 계산하여 Redis에 쓰는(Write) 역할만 전담하는 별도의 EC2 인스턴스.
  • Redis 서버: 모든 서버가 공유하는 중앙 캐시 저장소.

최종 결과

캐시 쓰기 서버를 분리하고 분산 캐시(Redis)를 도입한 결과, 메인 애플리케이션 서버의 메모리 점유율을 항상 200MB 이하로 안정적으로 유지할 수 있었습니다. 이를 통해 배포 안정성을 확보하고 OOM 장애 위험을 완전히 해소했습니다.

기능 특성상 유저는 지금 당장 태그가 생성되지 않더라도 태그가 생성되기만 하면 되므로, 동기방식에서 비동기로 전환하는 것을 꾀했습니다.

3단계: 새로운 문제 발견 - 시스템 안정성 위협

비동기 처리로 성능은 개선되었지만, 외부 API(OpenAI, Claude)의 갑작스러운 장애로 인해 TIL 태그 및 레퍼런스 데이터가 유실되는 치명적인 문제가 발생하여 서비스 제공에 영향을 주었습니다.

외부 시스템의 일회성 장애로 인한 데이터 정합성 훼손이 주요 이슈였습니다. 성능 개선은 했지만, 시스템 안정성 측면에서 새로운 도전과제가 생겼습니다.

4단계: Failover 및 재시도 큐로 시스템 안정성 확보

🔄 4-1: Chain of Responsibility 패턴 기반 Failover 구조
📊 문제 상황 분석

외부 API(OpenAI, Claude)의 갑작스러운 장애로 인해 TIL 태그 및 레퍼런스 데이터가 유실되는 치명적인 문제가 발생했습니다. 단일 외부 시스템에 의존하는 구조는 99.9% 가용성 요구사항을 충족할 수 없었습니다.

💡 의사결정 과정

외부 시스템 장애 시에도 서비스 연속성을 보장하기 위해 Chain of Responsibility 패턴을 적용했습니다. 실패 시 다음 외부 시스템을 순차적으로 호출하는 구조로 설계하여 단일 실패 지점을 제거했습니다.

✅ 결과 및 학습

외부 API 장애 시에도 99.9% 가용성을 확보하여 사용자 데이터 유실을 방지했습니다. 다중 외부 시스템 활용으로 서비스 안정성이 크게 향상되었습니다.

⚡ 4-2: 재시도 큐를 통한 최종 일관성 보장
📊 새로운 문제 발견

Failover 구조로도 모든 외부 시스템이 동시에 장애가 발생할 수 있었습니다. 이 경우 사용자 데이터가 영구적으로 유실될 위험이 있었고, 일시적 장애 후 복구 시에도 데이터 정합성을 보장할 수 없었습니다.

💡 의사결정 과정

모든 외부 시스템이 실패할 경우를 대비하여 스케줄링 기반 재시도 큐 재처리 아키텍처를 설계했습니다. 실패한 이벤트를 큐에 저장하고 @Scheduled 기반의 재시도 로직으로 최종적 일관성(Eventual Consistency)을 보장했습니다.

✅ 결과

외부 시스템의 일시적 장애에도 데이터 유실을 완전히 방지하고, 시스템 복구 후 자동으로 데이터 정합성을 복원할 수 있게 되었습니다.

이를 해결하고자 사용자 테스트를 주도하여 정성적 데이터를 확보했고, 이를 근거로 팀의 개발 목표를 핵심 기능 및 유저 중심으로 재설정했습니다.


해당 경험을 통해 팀 전체가 사용자의 목소리에 기반하여 움직이는 문화를 구축하며 팀의 원동력이 되었고

불필요한 기능이 아닌 사용자에게 필요한 기능을 빠르게 제공하는 목표에만 집중할 수 있었습니다.
